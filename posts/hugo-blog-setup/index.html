끝까지 파고드는 기록

처음에는 그냥 미션이었다.

강사님이 Hugo와 GitHub으로 블로그를 만들어 보라고 했고, 나는 시키는 대로 따라했다. 그런데 따라하다 보니 자꾸 의문이 생겼다. 이게 왜 이렇게 작동하지? 이 구조가 최선인가? 결국 구조를 이해하고 나서 직접 개선하는 데까지 이어졌다.

이 글은 그 과정을 처음부터 기록한 것이다.

시작: 강사님 미션
#

강사님이 준 미션은 다음과 같았다.

  1. GitHub 회원가입
  2. Git 설치 (개인 컴퓨터)
  3. Hugo 설치
  4. [githubID].github.io repo 생성 (Public)
  5. 블로그 repo 생성 (Private)
  6. 두 repo 연동 — git submodule 개념

처음 보는 단어들의 조합이었다. Git, GitHub, Hugo, submodule. 각각이 무엇인지, 왜 이렇게 구성하는지 모른 채로 따라가기 시작했다.

핵심 도구 4가지
#

미션을 이해하기 전에 각 도구가 무엇인지 먼저 정리해야 했다.

Git — 버전 관리 도구
#

내 컴퓨터에 설치하는 프로그램이다. 파일의 변경 이력을 추적하고, 로컬 컴퓨터와 GitHub 사이에서 파일을 주고받는 역할을 한다.

비유: 타임머신 + 택배 서비스. 언제든 과거 상태로 돌아갈 수 있고, 파일을 원격 서버로 보낼 수 있다.

GitHub — 클라우드 저장소 + 웹서버
#

Git 저장소를 인터넷에 올려두는 플랫폼이다. 여기서 핵심은 GitHub Pages 기능인데, 저장소 안의 HTML 파일을 그대로 웹사이트로 호스팅해준다. [ID].github.io 형식의 이름으로 저장소를 만들면 별도 설정 없이 연결된다.

비유: 클라우드 창고 + 무료 웹서버.

Hugo — 정적 사이트 생성기
#

Markdown(.md) 파일을 읽어서 HTML/CSS/JS로 변환해주는 도구다. 글 내용(md)과 테마(디자인)를 조합해 완성된 웹페이지를 만들어낸다. 글이 100개여도 1~2초 안에 전체 빌드가 끝날 만큼 빠르다.

비유: 출판 기계. 원고(md)와 디자인 틀(테마)을 넣으면 완성된 책(HTML)이 나온다.

Git Submodule — 저장소 간 연결
#

A 저장소 안에 B 저장소를 포함시키는 Git 기능이다. 폴더처럼 보이지만 실제로는 별개의 저장소와 연결되어 있어서, 해당 폴더 안에서 git push하면 연결된 저장소로 바로 올라간다.

비유: 바로가기 아이콘. 겉으로는 폴더처럼 보이지만 실제로는 다른 곳을 가리킨다.

기본 구조: 두 개의 저장소
#

강사님 미션의 핵심은 저장소를 두 개 운영하는 것이었다.

블로그 repo (Private)          [ID].github.io repo (Public)
├── content/posts/*.md         ├── index.html
├── themes/PaperMod/ ←submodule ├── css/
├── config.toml                └── posts/
└── public/ ←────submodule──────→    └── my-article/index.html
  • Private repo: md 원고, Hugo 설정, 테마. 작업 공간.
  • Public repo: Hugo가 만든 HTML 결과물. GitHub Pages가 이걸 읽어 웹에 띄운다.

public/ 폴더가 submodule로 [ID].github.io repo와 연결되어 있는 것이 핵심이다. 덕분에 public/ 안에서 push하면 자동으로 올바른 저장소에 올라간다.

기본 구조의 작동 흐름
#

  1. md 파일 작성 — content/posts/ 안에 글 작성
  2. hugo build 실행 — 로컬 Hugo가 전체 md를 읽어 HTML로 변환, public/에 저장
  3. cd public && git push — submodule 덕분에 [ID].github.io repo에 HTML 업로드
  4. cd .. && git push — 블로그 루트에서 md 원고를 Private repo에 백업
  5. GitHub Pages 자동 감지 — Public repo에 변경이 생기면 1~2분 안에 웹사이트 갱신
# 매번 글 올릴 때마다 반복하는 작업
$ hugo build
$ cd public
$ git add . && git commit -m "새 글 추가"
$ git push origin main
$ cd ..
$ git add . && git commit -m "원고 백업"
$ git push origin main

구조 자체는 논리적이었다. 하지만 직접 해보기도 전에 문제가 보였다.

문제 발견: 두 가지 비효율
#

문제 1. Hugo 로컬 설치
#

나는 컴퓨터를 두 대 사용한다. 강의실 컴퓨터와 집 컴퓨터. 기본 구조대로라면 두 컴퓨터 모두에 Hugo를 설치하고 환경을 세팅해야 한다. 컴퓨터가 바뀌거나 새로 포맷하면 또 처음부터 다시 세팅이다.

글을 쓰기 위해 매번 환경 의존성을 관리해야 한다는 게 근본적으로 불편했다.

문제 2. push 2번
#

글 하나 올릴 때마다 push를 두 번 해야 한다. public/ 에서 한 번, 블로그 루트에서 한 번. 매번 디렉토리를 이동하면서 두 번씩 커밋하고 push하는 과정이 반복된다.

한 번의 작업이 두 번의 명령 묶음으로 나뉘어 있다는 것 자체가 개선 가능한 구조로 보였다.

해결: GitHub Actions로 자동화
#

두 문제의 공통 원인은 Hugo 빌드를 내 컴퓨터에서 한다는 것이었다. 빌드를 내 컴퓨터가 아닌 GitHub 서버에서 처리하면 두 문제가 동시에 해결된다.

그 방법이 GitHub Actions다.

개선된 구조
#

블로그 repo (Private)
├── content/posts/*.md
├── themes/PaperMod/ ←submodule
├── config.toml
└── .github/workflows/deploy.yml  ← 자동화 설정 파일

public/ 폴더가 사라졌다. Hugo 빌드는 GitHub 서버가 하고, 결과물은 서버가 직접 Public repo에 push한다.

개선된 구조의 작동 흐름
#

  1. md 파일 작성 후 git push 1번 — 이게 전부다
  2. GitHub Actions 자동 실행 — Private repo에 push 감지 → Ubuntu 가상환경 생성 → Hugo 임시 설치 → hugo build 실행
  3. 자동 push — 빌드 결과물을 PAT(Personal Access Token)를 사용해 [ID].github.io repo에 직접 push
  4. GitHub Pages 자동 배포 — 1~2분 안에 웹사이트 갱신
# 개선 후: 이게 전부
$ git add . && git commit -m "새 글 작성"
$ git push origin main

PAT(Personal Access Token)가 필요한 이유
#

GitHub Actions는 다른 저장소에 push할 권한이 없다. PAT는 “이 작업은 나(저장소 주인)가 허가한 것이다"를 증명하는 열쇠다. 한 번 발급해서 저장소 설정에 등록해두면 이후 자동으로 사용된다.

두 구조 비교
#

항목기본 구조개선된 구조
Hugo 설치내 컴퓨터에 필요불필요
push 횟수2번1번
빌드 위치내 컴퓨터GitHub 서버
어느 기기서든 작성Hugo 설치된 기기만어느 기기서든 가능
로컬 미리보기가능 (hugo server)불가 (push 후 확인)
초기 설정 난이도비교적 단순PAT + yml 파일 필요
오류 확인터미널에서 바로GitHub Actions 로그 확인

개선된 구조도 단점이 있다. 로컬 미리보기가 안 되고, 초기 설정에 yml 파일 작성이 필요하다. 그러나 두 컴퓨터를 오가며 사용하는 나에게는 환경 독립성이 더 중요했다.

마치며
#

강사님 미션을 그대로 따라가지 않고 구조를 먼저 이해하려 했던 것이 결과적으로 좋았다. 왜 이렇게 구성됐는지 알고 나니 어디를 바꾸면 되는지도 보였다.

기술을 배울 때 “어떻게 쓰는가"보다 “왜 이렇게 만들어졌는가"를 먼저 묻는 습관이 여기서도 유효했다.
