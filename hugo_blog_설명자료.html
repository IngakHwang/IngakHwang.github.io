Part 01
핵심 도구 4가지
🗂
Git
버전 관리 도구
내 컴퓨터에 설치하는 프로그램. 파일의 변경 이력을 추적하고, 로컬 컴퓨터와 GitHub 사이에서 파일을 주고받는 역할을 한다.
비유: 타임머신 + 택배 서비스. 언제든 과거로 돌아갈 수 있고, 파일을 원격 서버로 보낼 수 있다.
☁️
GitHub
클라우드 저장소 + 웹서버
Git 저장소를 인터넷에 올려두는 플랫폼. GitHub Pages 기능을 통해 저장소 안의 HTML 파일을 웹사이트로 호스팅해준다.
비유: 클라우드 창고 + 무료 웹서버. 파일을 올려두면 전 세계에서 접근 가능하다.
⚙️
Hugo
정적 사이트 생성기
Markdown(.md) 파일을 읽어서 HTML/CSS/JS로 변환해주는 도구. 테마(디자인)와 글 내용(md)을 조합해서 완성된 웹페이지를 만들어낸다. 글이 100개여도 1~2초 안에 전체 빌드가 끝날 만큼 빠르다.
비유: 출판 기계. 원고(md)와 디자인 틀(테마)을 넣으면 완성된 책(HTML)이 나온다.
🔗
Git Submodule
저장소 간 연결 기술
A 저장소 안에 B 저장소를 포함시키는 Git 기능. 폴더처럼 보이지만 실제로는 별개의 저장소와 연결되어 있어, 해당 폴더 안에서 push하면 연결된 저장소로 바로 올라간다.
비유: 바로가기 아이콘. 겉으로는 폴더처럼 보이지만 실제로는 다른 곳을 가리킨다.
Part 02
강사님 미션: 기본 구조와 작동 방식

강사님이 제시한 구조는 로컬에 Hugo를 설치하고, 두 개의 저장소를 운영하는 방식이다.

저장소 구조
블로그 repo (Private) — 원고 보관
├── content/posts/my-article.md ← 글 원고
├── themes/PaperMod/ submodule ← 테마
├── config.toml ← Hugo 설정
└── public/ submodule ← 빌드 결과물 (아래 repo와 연결)
↕ submodule 연결
[ID].github.io repo (Public) — 웹사이트 배포
├── index.html
├── css/style.css
└── posts/my-article/index.html ← Hugo가 만든 HTML

submodule이 핵심 역할을 하는 이유: public/ 폴더가 [ID].github.io repo와 연결되어 있기 때문에, public/ 안에서 push하면 자동으로 올바른 저장소에 올라간다. 없었다면 빌드 결과물을 매번 수동으로 복사해서 다른 저장소에 따로 push해야 했을 것이다.

기본 구조 작동 흐름

1
md 파일 작성 직접
content/posts/ 안에 Markdown으로 글을 작성한다.
↓
2
hugo build 실행 직접
로컬에 설치된 Hugo가 md 파일 전체를 읽고, 테마를 입혀 HTML/CSS/JS로 변환한다. 결과물은 public/ 폴더에 저장된다. (항상 전체 빌드)
↓
3
public/ 에서 git push 직접 (submodule 작동)
cd public 후 git push → submodule 덕분에 [ID].github.io repo에 HTML이 업로드된다.
$ cd public
$ git add . && git commit -m "새 글 추가"
$ git push origin main
↓
4
블로그 repo에 git push 직접 (원고 백업)
cd .. 후 git push → md 원고를 Private repo에 백업한다. 웹사이트 배포와는 무관한 작업이다.
↓
5
GitHub Pages 자동 감지 및 배포 자동
[ID].github.io repo에 변경이 감지되면 GitHub Pages가 1~2분 안에 웹사이트를 자동 갱신한다.
⚡ 기본 구조의 문제점
문제 1. Hugo를 내 컴퓨터에 반드시 설치해야 한다. 컴퓨터가 바뀌면 다시 환경을 세팅해야 한다.

문제 2. push를 2번 해야 한다. public/ 에서 한 번, 블로그 루트에서 한 번 — 글 하나 올릴 때마다 반복되는 작업이다.
Part 03
개선된 구조: GitHub Actions 자동화

위 두 가지 문제를 해결하기 위해 고안한 구조. Hugo 빌드를 내 컴퓨터가 아닌 GitHub 서버에서 처리하도록 자동화한다.

개선된 저장소 구조
블로그 repo (Private) — 원고 + 자동화 설정
├── content/posts/my-article.md ← 글 원고
├── themes/PaperMod/ submodule ← 테마
├── config.toml
└── .github/workflows/deploy.yml ← GitHub Actions 설정 파일
✦ public/ 폴더가 로컬에 없다. GitHub Actions 서버가 빌드 후 직접 아래 repo에 push한다.
[ID].github.io repo (Public) — 빌드 결과물 자동 수신
├── index.html
└── posts/my-article/index.html

PAT(Personal Access Token)는 GitHub Actions가 Private repo에서 빌드한 결과를 Public repo에 push할 때 필요한 권한 열쇠다. 한 번 발급해서 저장소 설정에 등록해두면 이후 자동으로 사용된다.

개선된 구조 작동 흐름

1
md 파일 작성 후 git push 1번 직접
글을 쓰고 Private repo에 push한다. 이게 전부다. Hugo 빌드, public/ 폴더 이동, 두 번째 push — 전부 사라진다.
$ git add . && git commit -m "새 글 작성"
$ git push origin main
← 이게 끝. 나머지는 자동
↓
2
GitHub Actions 자동 실행 자동
Private repo에 push 감지 → GitHub 서버에 가상환경(Ubuntu) 생성 → Hugo 임시 설치 → md 파일 전체 가져오기 → hugo build 실행 → HTML 생성
↓
3
Public repo에 자동 push 자동
GitHub Actions가 PAT를 사용해서 빌드 결과물을 [ID].github.io repo에 직접 push한다. 기존에 내가 직접 하던 "cd public → git push" 과정을 서버가 대신한다.
↓
4
GitHub Pages 자동 배포 자동
Public repo 변경 감지 → 1~2분 안에 웹사이트 갱신 완료
기본 구조 vs 개선된 구조 비교

항목 기본 구조 (강사님 미션) 개선된 구조 (GitHub Actions)
Hugo 설치 내 컴퓨터에 필요 불필요 (서버가 처리)
push 횟수 2번 (public/ + 블로그 루트) 1번 (블로그 루트만)
빌드 위치 내 컴퓨터 GitHub 서버
어느 기기서든 작성 Hugo 설치된 기기만 어느 기기서든 가능
로컬 미리보기 가능 (hugo server) 불가 (push 후 확인)
초기 설정 난이도 비교적 단순 PAT + yml 파일 설정 필요
오류 발생 시 터미널에서 바로 확인 GitHub Actions 로그 확인 필요
✦ 개선된 구조의 장점
Hugo 로컬 설치 불필요 — 환경에 독립적
push 1번으로 배포까지 자동 완성
GitHub 웹에서 md 편집 후 push만 해도 배포됨
컴퓨터가 바뀌어도 세팅 불필요
✦ 개선된 구조의 단점
PAT 발급 및 주기적 갱신 필요
로컬 미리보기 불가 — push 후에야 확인
yml 설정 파일 작성 필요 (초기 1회)
빌드 실패 시 GitHub 로그에서 디버깅
